# Project Principles

## Назначение

Этот документ фиксирует архитектурные и инженерные принципы проекта. Они обязательны при проектировании базы данных, API, интерфейса, интеграций и бизнес-логики.

## 1. Single Source of Truth

Все основные данные существуют в одном месте: PostgreSQL. Внешние сервисы могут быть источником импорта или каналом доставки, но не должны становиться главным хранилищем бизнес-данных.

## 2. Mobile First

Любой пользовательский сценарий сначала проектируется для телефона. Desktop-интерфейс может расширять возможности просмотра и управления, но не должен быть единственным способом выполнить ключевую рабочую операцию.

## 3. Security First

Клиентское приложение не считается доверенной средой. Права доступа, бизнес-ограничения и критичные действия проверяются на стороне сервера, базы данных или Edge Functions.

## 4. Immutable History

Исторические данные не переписываются прямыми изменениями. Закрытые смены, сформированные snapshots, payroll-расчеты и audit log должны сохранять историю так, чтобы можно было восстановить состояние на момент события.

## 5. Role Based Access Control

Доступ строится на ролях и области ответственности пользователя. Frontend может скрывать недоступные действия для удобства, но окончательное решение о доступе принимает backend и база данных.

## 6. Auditability

Критичные изменения должны оставлять след: кто выполнил действие, когда оно выполнено, что изменилось и по какой причине.

## 7. Scalability

Архитектура должна поддерживать рост количества магазинов, сотрудников, смен, задач и продаж без переписывания ключевых модулей.

## 8. Documentation Driven Development

Сначала фиксируется правило или техническое решение в документации, затем оно реализуется в коде. Если код меняет бизнес-логику, связанная документация обновляется в той же задаче.

## 9. No Silent Assumptions

Если бизнес-правило не определено, оно помечается как Draft или выносится на уточнение.

## 10. Production Readiness

Код, миграции и интеграции проектируются как основа production-системы. Временные решения допустимы только как локальные инструменты разработки и не должны попадать в production-процессы.
