# Project Overview

## 1. Общая информация

### Название проекта

Retail Staff Management System (рабочее название)

### Назначение

Система предназначена для управления сетью розничных магазинов спортивного питания. Она заменяет разрозненные Google Таблицы, переписки в Telegram и ручные расчеты единым web-приложением с централизованной базой данных.

## 2. Цель проекта

Создать современное web-приложение в формате PWA и Telegram Mini App, которое автоматизирует ежедневную работу сотрудников, управляющих, проверяющих и администрации.

Ключевая цель: все рабочие процессы должны выполняться в одной системе, а PostgreSQL должен быть единственным источником истины.

## 3. Форматы запуска

- Telegram Mini App.
- Web-браузер.
- PWA с установкой на главный экран телефона.

## 4. Платформы

### Mobile

- iPhone.
- Android.

Приложение проектируется по принципу Mobile First.

### Desktop

Поддерживаются современные браузеры: Google Chrome, Microsoft Edge, Safari, Firefox.

## 5. Масштаб запуска

На момент запуска система должна поддерживать около 25 магазинов, около 15 городов и дальнейшее расширение сети без изменения архитектуры.

## 6. Основная бизнес-модель

Все магазины принадлежат одной компании и работают по одинаковой бизнес-модели.

Каждый магазин имеет собственный график работы, персонал, KPI, показатели продаж, одну или несколько касс, историю смен, задач, проверок и расчетов.

## 7. Основные бизнес-сущности

- Магазин.
- Сотрудник.
- Смена.
- График.
- Задача.
- KPI.
- Зарплата.
- Проверка магазина.
- Чеклист.
- OFD данные.
- Уведомления.
- Пользователь.
- Роль.
- Бонус.
- Штраф.
- Файл.
- Запись аудита.

Главной сущностью системы является **смена**. Большинство процессов строится вокруг жизненного цикла смены.

## 8. Основные процессы

### Управление сотрудниками

Создание, блокировка, увольнение, изменение ролей и перевод между магазинами.

### Управление графиком

Создание и изменение смен, учет выходных, отпусков и больничных.

### Работа менеджера

Открытие смены, прохождение чеклиста, выполнение задач, просмотр KPI, зарплаты и графика, закрытие смены.

### Работа проверяющего

Проведение проверки магазина, оценка менеджеров и просмотр истории проверок.

### Работа управляющего

Управление сотрудниками, графиком, задачами, аналитикой, KPI, зарплатами и чеклистами.

### Работа супер-администратора

Супер-администратор имеет полный контроль над системой.

## 9. Источники данных

Основной источник данных: сама система.

Дополнительные интеграции: Telegram Bot API, СБИС OFD, Google Sheets для экспорта данных.

Google Sheets не являются источником истины. Все основные данные хранятся в PostgreSQL.

## 10. Технологический стек

Frontend: Next.js, React, TypeScript, TailwindCSS, Zustand.

Backend: Supabase, PostgreSQL, Edge Functions.

Storage: Supabase Storage.

Notifications: Telegram Bot API, Web Push API.

Hosting: Vercel для frontend, Supabase для backend.

## 11. Общие требования

В системе не допускается использование временных решений как production-логики, тестовых заглушек в production, ручное изменение KPI без регламентированной записи, ручное изменение автоматически рассчитанной зарплаты без записи в журнал изменений и хранение источника истины вне PostgreSQL.

Любое изменение бизнес-логики требует обновления документации.
